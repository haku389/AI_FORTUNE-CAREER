## 2. スコア計算ロジック（precise-scoring.ts）

### 出力される4つのスコア

| スコア | 範囲 | 説明 |
|--------|------|------|
| score_timing | 30〜97 | 転職タイミングスコア（urgency量から算出） |
| score_readiness | 30〜97 | 転職準備スコア（市場価値・他者評価・行動起点） |
| score_market | 30〜97 | 市場相性スコア（独立志向・年収志向） |
| score_total | 30〜97 | 上記3つの単純平均 |

---

### 2-1. score_timing（タイミングスコア）

**ステップ1: urgency合計**

全24問のうち `urgency_add` を持つ回答を全て合計する。

urgency_addが設定されている質問：

| Q（index） | 最大urgency_add |
|-----------|----------------|
| Q1(0) | 4 |
| Q2(1) | 2 |
| Q3(2) | 3 |
| Q4(3) | 3（在籍7年以上） |
| Q14(13) | 1 |
| Q15(14) | 1 |
| Q16(15) | 2（self_doubt） |
| Q20(19) | 2（independent） |
| Q24(23) | 3 |

理論最大合計: 約22（コード内 `MAX_URGENCY = 22` で設定）

**ステップ2: スコア変換**

```
score_timing = clamp(round((urgency / 22) * 67 + 30), 30, 97)
```

| urgency | score_timing |
|---------|-------------|
| 0 | 30 |
| 5 | 45 |
| 10 | 60 |
| 15 | 76 |
| 22 | 97 |

**ステップ3: timing区分判定**

| urgency | timing | バッジ |
|---------|--------|--------|
| ≥15 | now | ⚡ 今すぐが転職の好機 |
| 10〜14 | 3m | 🌕 3ヶ月以内が転職好機 |
| 5〜9 | 6m | 🌙 半年後に転職の波が来る |
| 0〜4 | wait | 🌱 今は充電・準備の時期 |

### 2-4. score_total（総合スコア）

```
score_total = clamp(round((score_timing + score_readiness + score_market) / 3), 30, 97)
```

3つの算術平均。重みは均等。

## 3. 職種マッチングロジック（jobMatch.ts）

### 入力

| 変数 | 取得元 |
|------|--------|
| mbti | Q5(index 4)の選択 → NT/NF/SJ/SP/unknown |
| motivation | Q6(index 5)の選択 → contribution/learning/creation/achievement |
| roles | Q13(index 12)の複数選択 → role文字列の配列 |
| industries | Q12(index 11)の複数選択 → industry文字列の配列 |

### 職種スコア計算

14職種それぞれに対して:

```
score[job] = 20（ベース）
           + Σ ROLE_JOB_SCORES[role][job]     // 選択した各roleの寄与
           + MBTI_JOB_BONUS[mbti][job]         // MBTI補正
           + MOTIVATION_BONUS[motivation][job]  // モチベーション補正
```

### TOP3スコアの表示変換（68〜92%）

1. 全14職種のraw scoreを計算し降順ソート
2. TOP3を抽出
3. 比例変換: `proportional = round(((rawScore - minRaw) / rawRange) * 24 + 68)`
4. ランク別最低値保証で表示スコア確定:

```
RANK_FLOORS = [92, 80, 68]  // rank1, rank2, rank3 の最低値

rank1: score = 92（固定）
rank2: score = max(80, min(prevScore - 6, proportional))
rank3: score = max(68, min(prevScore - 6, proportional))
```

> rank1は常に92%固定。rank2は80〜86%、rank3は68〜74%の範囲に収まる。

## 4. 現状の問題点・設計上の課題

### 4-1. スコアに使われていない質問が多い

| 質問 | 使用状況 |
|------|---------|
| Q7 ストレス反応 | 記録のみ（スコア・マッチング未使用） |
| Q8 リーダー像 | 記録のみ |
| Q9 コミュニケーション | 記録のみ |
| Q10 意思決定 | 記録のみ |
| Q11 現業界 | 記録のみ |
| Q17 働き方 | 記録のみ |
| Q18 組織規模 | 記録のみ |
| Q19 WLB | 記録のみ |

→ **24問中8問がスコア・マッチング計算に一切影響しない**

### 4-2. score_readinessの分散が小さい

- Q22（他者評価）の`externalViewScore`が73〜83の10pt幅しかない
- 実質的にreadinessはほぼ市場価値(Q21)と行動起点(Q23)で決まる

### 4-3. score_marketの設計意図が不明瞭

- 「独立志向」と「年収志向」の平均というのは相関が弱い
- 高独立志向（87）+ 高年収志向（87）= 87点 → 「市場相性が高い」の意味が曖昧

### 4-4. timing区分の閾値が厳しい

- urgency ≥15（22満点中）でようやく"今すぐ"判定
- Q1で+4を取っても残り11点分が必要 → 多くのユーザーが「3m」「6m」判定になりがち

### 4-5. score_total が3つの平均なのでメリハリが弱い

- timing(30〜97) + readiness(30〜97) + market(30〜97) の単純平均
- 各スコアの性質が違う（タイミング vs 準備度 vs 志向性）のに同等重みは疑問

### 4-6. urgency_add が重複加算される懸念

- 複数選択(multi)のQ14でurgency_addを選択数分合計している
- 例：Q14でopt0とopt1を両方選んだ場合 urgency += 1 + 1 = 2（実際Q14はsingleだが設計上のリスク）
